## 结论

本地项目侧已经完成 Coze 调用所需的基础配置和适配修复：

- Coze API 地址已使用 `https://api.coze.cn/v1/workflow/run`
- `.env` 已配置工作流 ID 和本地 token
- 本地服务已可读取 Coze 配置
- 本地测试已通过

当前阻塞点主要在 Coze 平台工作流内部。只要工作流在 Coze 调试页能稳定返回 `code=0` 且输出结构符合下面约定，本地项目即可正常调用。

## 已解决的本地问题

### 1. Coze API 地址错误

问题：

旧配置使用过 `https://api.coze.cn/v3/workflows/run`，实测 Coze 返回接口不存在。

解决办法：

`.env` 使用：

```env
COZE_API_URL=https://api.coze.cn/v1/workflow/run
```

处理状态：已解决。

### 2. 切题工作流入参字段不一致

问题：

切题工作流导出包里预处理节点读取的是：

```js
data.text_content
```

而本地早期调用只传过：

```json
{
  "paper_text_data": {
    "content": "..."
  }
}
```

解决办法：

本地 `CozeService.execute_split()` 现在同时传：

```json
{
  "paper_text_data": {
    "content": "...",
    "text_content": "...",
    "paper_id": "A"
  }
}
```

处理状态：已解决。

### 3. Coze 调用缓存没有区分 workflow ID

问题：

本地缓存原来只按 `parameters` 做 key。如果两个不同工作流刚好传入相同参数，可能错误复用上一次结果。

解决办法：

缓存 key 已加入：

```json
{
  "workflow_id": "...",
  "parameters": {}
}
```

处理状态：已解决。

### 4. Coze 返回 JSON 字符串时解析不稳

问题：

Coze 的 LLM 节点有时会把结构化 JSON 放在字符串里返回，例如：

```json
"{\"plagiarism_details\": [...]}"
```

或循环输出里每一项都是 JSON 字符串。本地原来只识别对象数组，会漏掉这些结果。

解决办法：

查重和错字解析现在会递归尝试解析 JSON 字符串，并兼容带 ```json 代码块的输出。

处理状态：已解决。

### 5. Coze 查重结果标签误写成女娲知识库

问题：

走 Coze 查重时，报告里的历史库标签仍可能显示为“女娲知识库”。

解决办法：

现在会按实际服务类型显示：

- Coze：`Coze 知识库`
- Nuwa：`女娲知识库`

处理状态：已解决。

### 6. Coze 导出包可能被误提交

问题：

`data/coze_exports/` 内包含 Coze 工作流导出 YAML，其中可能有空间 ID、知识库 ID、节点配置等信息。

解决办法：

`.gitignore` 已加入：

```gitignore
data/coze_exports/
```

处理状态：已解决。

### 7. Coze 适配缺少回归测试

问题：

之前没有测试覆盖 Coze 切题入参、缓存隔离和 JSON 字符串查重结果解析。

解决办法：

已新增测试覆盖：

- 切题调用传文本内容
- 切题 payload 同时包含 `content` 和 `text_content`
- 缓存按 workflow ID 隔离
- 查重结果中的 JSON 字符串可被解析

处理状态：已解决。

## 仍需在 Coze 平台修复的问题

### 1. 切题工作流仍返回 `code=5000`

现象：

即使本地传入 `text_content`，切题工作流仍返回 Coze 内部错误。

建议在 Coze 工作流中检查：

- 工作流是否已发布
- 开始节点入参是否为 `paper_text_data`
- 文本预处理代码是否写成兼容形式：

```js
function main(args) {
  const data = args.input_obj || {};
  let text = data.text_content || data.content || "";
  let paper_id = data.paper_id || "UNKNOWN";
  let subject = data.subject || "UNKNOWN";

  text = text.replace(/第\s*\d+\s*页/g, "");
  text = text.replace(/^\s*-\s*\d+\s*-\s*$/gm, "");
  text = text.replace(/\n{3,}/g, "\n\n");

  return {
    clean_text: text,
    paper_id,
    subject
  };
}
```

先让结束节点稳定返回：

```json
{
  "split_result": {
    "paper_id": "A",
    "subject": "chinese",
    "questions": [
      {
        "question_id": "A-1",
        "question_no": "1",
        "order": 1,
        "content": "题目内容"
      }
    ]
  }
}
```

### 2. 查重工作流仍返回 `code=5000` 或超时

现象：

综合查重工作流能通过鉴权，但运行阶段返回内部错误或超时。

建议在 Coze 工作流中检查：

- 知识库检索 Query 不要传整个题目对象，改传题干文本：

```text
item.content
```

- LLM prompt 里变量名要和节点输入一致。当前节点输入是 `kb_history`，prompt 不应使用不存在的 `history_questions`。
- 循环输出如果取的是 `reasoning_content`，整合节点要先解析 JSON 字符串：

```js
const plagiarisms = (args.plagiarism_data || [])
  .map(item => {
    if (typeof item === "string") {
      try {
        return JSON.parse(item);
      } catch (e) {
        return null;
      }
    }
    return item;
  })
  .filter(Boolean);
```

- 最小可跑时，先返回空数组也可以：

```json
{
  "output_report": {
    "error_checklist": [],
    "plagiarism_details": []
  }
}
```

先保证工作流 `code=0`，再逐步调效果。

# EchoPaper 前端改造方案

版本：v1.0  
日期：2026-05-10

## 一、结论

EchoPaper 前端**建议做中到大改**，但**不建议重做技术栈**。

更具体地说：

- **不需要**改成 React / Vue 前后端分离。
- **需要**重构当前 Jinja2 模板结构、样式组织方式和页面信息架构。
- **优先级最高**的页面是 `report.html`，其次是 `report_snapshot.html`，然后才是 `index.html`、`reports.html`、`history_bank.html`、`agent_jobs.html`。

当前最核心的问题不是“页面不能用”，而是：

- 页面样式大量内联在模板里，后续维护成本会持续升高。
- 报告页信息量已经很大，但缺少清晰的信息分层和风险优先排序。
- 实时报告页、历史快照页、列表页不是同一套视觉系统，后续会越来越散。
- 审查系统的核心体验应该是“快速发现高风险并做人工复核”，当前前端还没有围绕这个目标展开。

因此，前端改造目标不是单纯“换个更好看的皮”，而是把 EchoPaper 从“能展示结果的 MVP 页面”升级成“适合老师或教务连续使用的审查工作台”。

## 二、现状问题

### 2.1 模板结构问题

当前模板文件主要包括：

- `templates/index.html`
- `templates/report.html`
- `templates/report_snapshot.html`
- `templates/reports.html`
- `templates/history_bank.html`
- `templates/agent_jobs.html`

当前问题：

- 每个页面都带大量内联 CSS。
- 页面之间没有公共 `base.html`。
- 没有统一的 `partials` 分层。
- 公共元素如按钮、badge、表格、空状态、顶部导航没有统一抽象。
- `report.html` 已经比较大，后续继续堆功能会非常难改。

### 2.2 报告页信息架构问题

`report.html` 当前已经包含：

- 总览指标
- 上传试卷
- 疑似原题 / 同源题
- 解析风险提示
- 切题复核提示
- 复杂题复核提示
- 重复题明细
- 错字问题
- 双链路状态

问题不在于信息不够，而在于：

- 重要信息没有形成“风险优先”的浏览顺序。
- 用户进入报告后，第一眼仍然需要自己判断先看哪里。
- 同类信息之间的层级不够清晰。
- 长表格和长区块较多，连续审查时容易疲劳。

### 2.3 视觉系统问题

当前页面风格并不统一：

- `index.html` 偏暖色卡片风格。
- `report.html` 是另一套更复杂的暖色报告风格。
- `report_snapshot.html` 偏后台管理面板风格。

这会导致：

- 系统不像一个完整产品，更像几个拼起来的页面。
- 用户在上传、查看报告、查看历史快照时体验割裂。
- 以后加新页时没有明确可复用的视觉基线。

### 2.4 审查交互问题

EchoPaper 的核心不是“展示数据”，而是“辅助人工复核”。  
因此前端应该优先支持：

- 快速定位高风险项
- 快速筛选需要复核的题
- 快速修改人工复核状态
- 快速在实时报告和历史报告之间切换

当前这些能力还不够突出。

## 三、改造原则

### 3.1 不改技术栈

继续使用：

- `FastAPI`
- `Jinja2`
- 原生 CSS
- 少量原生 JS

原因：

- 当前系统以后端渲染为主，适合审查报告类页面。
- 页面状态复杂度还没高到必须引入 SPA。
- 当前重点是结构治理和产品体验，不是框架迁移。

### 3.2 先做结构治理，再做视觉翻新

优先级顺序应为：

1. 模板拆分
2. 公共样式抽离
3. 信息架构重排
4. 审查交互增强
5. 视觉精修

不能直接先“改颜色、改按钮、改排版”，否则后面还是会回到一份越来越大的模板里继续堆代码。

### 3.3 风险优先

报告页必须围绕“先看高风险，再看明细”来组织。

因此页面顺序应优先展示：

- 解析高风险
- 复杂题复核
- 低置信度切题
- 疑似原题 / 同源题
- 高度重复 / 疑似重复

而不是把所有模块按开发顺序平铺。

### 3.4 一套视觉系统，多页复用

上传页、报告页、快照页、管理页应该属于同一产品，而不是几种风格混用。

建议建立统一设计基础：

- 颜色变量
- 字体层级
- 卡片样式
- 按钮样式
- Badge 样式
- 表格样式
- 空状态样式
- 响应式断点

## 四、目标结构

### 4.1 模板目录改造建议

建议把模板结构改为：

```text
templates/
├─ base.html
├─ index.html
├─ report.html
├─ report_snapshot.html
├─ reports.html
├─ history_bank.html
├─ agent_jobs.html

   ├─ layout/
   │  ├─ top_nav.html
   │  ├─ page_header.html
   │  └─ footer.html
   ├─ report/
   │  ├─ report_summary.html
   │  ├─ report_risk_nav.html
   │  ├─ report_parse_quality.html
   │  ├─ report_complex_questions.html
   │  ├─ report_question_quality.html
   │  ├─ report_same_source.html
   │  ├─ report_duplicate_rows.html
   │  ├─ report_spellcheck_rows.html
   │  └─ report_dual_run.html
   └─ shared/
      ├─ badges.html
      ├─ empty_state.html
      └─ export_actions.html
```

### 4.2 静态资源目录改造建议

建议新增：

```text
static/
├─ css/
│  ├─ tokens.css
│  ├─ base.css
│  ├─ layout.css
│  ├─ components.css
│  ├─ report.css
│  └─ admin.css
└─ js/
   ├─ report.js
   ├─ snapshot.js
   ├─ filters.js
   └─ export.js
```

说明：

- `tokens.css`：颜色、圆角、阴影、字号、间距变量
- `base.css`：基础 reset 和全局排版
- `layout.css`：容器、页头、导航、网格
- `components.css`：按钮、badge、表格、空态、表单
- `report.css`：审查报告特有布局
- `admin.css`：历史报告、题库、任务页等后台页风格

## 五、页面改造方案

### 5.1 上传页 `index.html`

当前定位：

- 单纯的上传表单页

建议升级为“系统首页 / 上传工作台”。

#### 改造目标

- 让用户一进来就知道系统能做什么
- 上传动作更明确
- 历史入口更清楚
- 为后续扩展保留空间

#### 建议版块

1. 顶部说明区
- 系统定位
- 当前支持范围
- 当前版本限制

2. 上传表单区
- 教师姓名
- 教师工号
- 科目
- A/B 卷上传
- 提交按钮

3. 快速入口区
- 查看历史报告
- 管理历史题库
- 查看 Agent 任务

4. 使用说明区
- 支持的文件类型
- OCR / 图片题说明
- 当前审查流程说明

5. 最近能力说明区
- 当前主链路能力
- 后续增强方向

#### 是否需要大改

上传页建议**中改**，不需要完全重画，但需要从“表单页”升级成“工作台首页”。

---

### 5.2 实时报告页 `report.html`

这是本次前端改造的最高优先级页面。

#### 改造目标

- 让用户快速识别高风险项
- 减少一次性看到过多信息的压力
- 提升人工复核效率
- 为后续新增模块保留稳定位置

#### 建议信息架构

建议按以下顺序重排：

1. 报告头部
- 教师信息
- 科目
- 上传文件
- 生成时间
- 导出按钮
- 返回入口

2. 总览指标区
- A 卷题目数
- B 卷题目数
- 高度重复
- 疑似重复
- 疑似原题
- 错字问题
- 历史题库命中
- 待人工确认

3. 风险导航区
- 高风险解析
- 复杂题复核
- 低置信度切题
- 疑似原题
- 重复题
- 错字问题
- 双链路状态

4. 风险优先区
- 解析风险提示
- 复杂题复核提示
- 切题复核提示
- 疑似原题 / 同源题

5. 主审查区
- 重复题明细
- 错字漏字清单
- 历史题库命中摘要

6. 对照信息区
- 双链路模块状态
- Agent 回填摘要

7. 附录区
- 上传试卷明细
- 系统诊断信息

#### 交互增强建议

建议新增：

- 区块锚点导航
- “只看高风险”筛选
- “只看 A 卷 / B 卷”筛选
- “只看待确认”筛选
- 长明细默认折叠，点击展开
- 表格支持局部搜索

#### 视觉建议

- 用明显的风险色区分 `高风险 / 需复核 / 提示`
- 把最需要老师点开的卡片做得更靠前、更突出
- 降低背景装饰复杂度，把视觉重点让给内容
- 报告页应优先“稳、清晰、耐看”，不要太花

#### 是否需要大改

报告页建议**大改**。

但这里的大改主要是：

- 结构大改
- 模板拆分
- 信息架构大改

不是把后端数据结构推翻重写。

---

### 5.3 历史快照页 `report_snapshot.html`

当前问题：

- 功能能用，但和实时报告页像两个系统

建议目标：

- 与实时报告页保持同一套视觉语言
- 明确区分“只读快照”与“实时审查”
- 保留导出、复核、查看历史记录能力

#### 改造建议

- 复用 `report.html` 的大部分布局结构
- 使用同一套卡片、表格、badge 样式
- 在顶部明确标识“历史快照 / 只读恢复报告”
- 把导出记录、快照时间、session 信息放到头部信息区

#### 是否需要大改

快照页建议**中到大改**，但应尽量建立在实时报告页改造成果之上。

---

### 5.4 历史报告列表页 `reports.html`

建议从“列表页”升级为“报告检索页”。

#### 建议增强

- 顶部筛选区固定
- 支持按教师、科目、关键词筛选
- 列表卡片化或更清晰的表格式摘要
- 每条记录显示：
  - 教师
  - 科目
  - 生成时间
  - 风险摘要
  - 导出次数
  - 最近导出格式

#### 目标

- 让用户能快速回看哪一份报告值得打开
- 不只是一个数据库记录表

---

### 5.5 历史题库页 `history_bank.html`

建议从“管理页”升级为“题库工作台”。

#### 建议增强

- 上传区独立
- 当前统计卡独立
- 最近重建任务独立
- 文件列表支持更清楚的筛选状态展示
- 删除按钮与危险操作样式统一

#### 重点

- 这个页面偏后台管理，不需要像报告页那样重内容，但必须清晰、稳、低认知负担

---

### 5.6 Agent 任务页 `agent_jobs.html`

建议统一到后台页风格，不再单独长成一页。

#### 建议增强

- 状态筛选
- 最近失败任务高亮
- 可直接跳到对应报告
- 可查看任务完成摘要

### 本轮代码组织逻辑

为了方便后续审查和继续开发，本轮前端代码按“基础层 -> 组件层 -> 页面层 -> 行为层”组织：

1. 基础样式层
- `static/css/tokens.css`：颜色、圆角、字号、容器宽度等设计变量
- `static/css/base.css`：基础排版和 reset
- `static/css/layout.css`：页面容器、顶部导航、页头、页脚

2. 组件样式层
- `static/css/components.css`：按钮、表格、badge、empty state、metric card、表单控件
- `static/css/report.css`：报告页专属布局
- `static/css/admin.css`：历史页、题库页、任务页等后台页样式

3. 页面模板层
- 所有页面统一继承 `base.html`
- 可复用结构优先放入 `partials/layout` 和 `partials/shared`
- 报告页高复杂区块拆入 `partials/report`，避免 `report.html` 再次变成超大单文件

4. 前端行为层
- `static/js/export.js`：统一处理 JSON/PDF 导出按钮
- `static/js/report.js`：实时报告页导出、Agent 轮询、人工复核保存
- `static/js/snapshot.js`：历史快照页导出、人工复核保存
- `static/js/filters.js`：历史题库后台重建任务轮询

## 八、验收标准

### 8.1 结构验收

- 所有页面不再以内联 CSS 为主
- 存在统一 `base.html`
- `report.html` 已拆为多个 partial
- 公共按钮、表格、badge、空状态有统一样式

### 8.2 报告页验收

- 用户进入报告后能在 5 秒内定位高风险区块
- 解析风险、复杂题、切题复核、疑似原题这四类内容位于前半屏或近前位置
- 重复题和错字问题支持更高效浏览
- 人工复核状态修改路径清晰

### 8.3 视觉验收

- 上传页、报告页、快照页、管理页属于同一套产品视觉风格
- 风险状态颜色统一
- 表格与卡片样式统一
- 空状态、按钮、导航风格统一
- 首页工作台已补齐说明区、快速入口和能力摘要，不再只是单表单页面

### 8.4 移动端验收

- 页面在手机宽度下不出现主要功能不可用
- 长表格有横向滚动或替代布局
- 顶部操作区不会挤坏
- 首页上传表单、快速入口、提交结果摘要在窄屏下可单列阅读和点击

## 十、最终建议

当前最合理的前端改造路线是：

1. **不迁移框架**
2. **先重构模板结构**
3. **优先大改报告页**
4. **再统一快照页和后台管理页**
5. **最后做视觉精修**

如果只做小修小补，短期能继续用，但后面每新增一个审查模块，前端都会越来越重、越来越难维护。  
如果按本方案推进，EchoPaper 前端会从“MVP 展示页面”升级成“可持续迭代的审查工作台”。

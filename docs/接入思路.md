# 接入思路

## 为什么先做这个 MVP

你现在缺两块关键能力：

1. 没有真实数据。
2. 女娲智能体工作流还没完成。

所以更稳的推进方式不是一上来做全链路，而是先把下面三件事固定下来：

- 教师端到底怎么操作
- 报告最终长什么样
- 后续各模块之间交换什么 JSON

这个仓库里的 MVP 就是在做这件事。

## MVP 的定位

它不是最终系统，也不是精度验证版。

它更像一个“业务骨架 + 接口占位”的演示版本：

- 前端把上传、处理、报告、人工确认跑通
- OCR 先用 mock 结构化题目代替
- 女娲先用规则引擎占位
- 历史库先用 mock 数据代替

这样你可以先和老师、产品、你朋友一起对齐：

- 查重报告结构对不对
- 人工复核流程顺不顺
- 女娲输出什么字段最方便接
- 历史库检索结果怎样展示更可信

## 推荐的真实落地顺序

### 1. 先做文科 PoC

先用语文 / 政治跑通：

- A/B 卷对比
- 历史库检索
- 错字漏字识别
- 在线报告生成

原因很简单：文科对公式 OCR 和图表识别依赖低，能更快验证“查重结果到底有没有用”。

### 2. 再接女娲工作流

建议女娲先只做错字、漏字、语病这条线，不要一开始就让它承担“是否原题”的最终裁决。

原因：

- 错字工作流边界清晰
- 输出更容易解释
- 比“是否原题”更容易先做稳定

建议输入：

```json
{
  "paperId": "A",
  "subject": "chinese",
  "customDictionary": ["价值规律", "二项式定理", "电势能"],
  "questions": [
    {
      "id": "A-2",
      "number": 2,
      "stem": "请分析诗句的表达效果"
    }
  ]
}
```

建议输出：

```json
{
  "paperId": "A",
  "findings": [
    {
      "questionId": "A-2",
      "snippet": "循序渐近",
      "suspectedIssue": "常用成语误写",
      "suggestion": "循序渐进",
      "confidence": 0.93
    }
  ]
}
```

### 3. 再接 OCR / 多模态解析

OCR 模块最好不要直接吐整页纯文本，而是吐结构化题目：

```json
{
  "paperId": "A",
  "subject": "math",
  "questions": [
    {
      "id": "A-1",
      "number": 1,
      "type": "函数题",
      "stem": "已知函数 f(x)=x^2-4x+3，求最小值",
      "options": [],
      "formulas": ["f(x)=x^2-4x+3"],
      "figures": [],
      "rawPage": 2
    }
  ]
}
```

这样后面可以拆成三路处理：

- 文本向量
- 公式向量
- 图表特征

### 4. 最后再做“是否原题”的大模型裁决

不要一上来把所有题都丢给大模型判。

更稳的方式是：

1. 先用向量检索召回候选题
2. 再用关键词 / 规则做一轮排序
3. 只把 Top-K 候选题交给大模型判断

这样成本和准确率都会更可控。

## 建议的数据流

### 前端

- 上传 PDF
- 选择科目
- 展示进度
- 展示报告
- 保存教师人工确认结果

### OCR Agent

- 输入 PDF
- 输出结构化试题 JSON

### 女娲 Agent

- 输入结构化试题
- 输出错字 / 语病 / 可疑表达

### 对比引擎

- 输入结构化试题
- 做 A 内部、B 内部、A/B、历史库四类比对
- 输出相似度、证据、候选题来源

### 报告模块

- 聚合 Dashboard
- 聚合错字清单
- 聚合逐题比对
- 落到教师人工确认区

## 你下一步最值得做的事

如果你现在要最快继续推进，我建议顺序是：

1. 用这个 MVP 先和使用方确认报告长相与交互流程
2. 跟你朋友对齐女娲工作流的输入输出 JSON
3. 先找一小批文科 PDF 手工整理成结构化样本
4. 用这些样本验证“90% 以上是否真能代表原题”
5. 再决定理科 OCR 和公式比对怎么做
